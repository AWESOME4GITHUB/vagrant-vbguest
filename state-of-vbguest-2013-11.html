current state of a Scratching-Your-Own-Itch-Project

2013-11-04 by Robert aka. @fnordfish.

Over two years ago I started to use Vagrant as a regular basis to build my development environments with. At this time, VirtualBox was very buggy (some will say, it still is) and thus updates where pretty frequent. After a while I got sick of manually updating the "GuestAdditons" in each and every box me and my coworkers where running manually and started a small plugin called vagrant-vbguest.

Thanks

But before going further, many thanks to all of you contributing (not only with code) to this project and for taking the time to help make vbguest better. Your help is really appreciated.

In the last months I didn't had much time to work on the plugin. And like many others, I started to use VMware Fusion as my main working environment. So the itches I scratched with building vbguest are pretty much gone.

Still, this project is important to me and I'm going to maintain and enhance it.

pretty messy

Lots of code (and certainly most of the ugly parts) is dedicated to provide backwards compatibility to old gem-install style Vagrant versions. There was some demand of non-client-side users who where stuck on old version of Vagrant and I wanted to keep them getting new features and bug fixes.

With latest versions of Vagrant, the plugin API got very stable. However, internals changed a lot in early 1.x builds â€“ particularly using the build-in SSH connection, and accessing a VM's ID.

Since the whole purpose of vbguest is to cope with version differences, I'm very aware of the fact, that people are not alway able to run the same, most recent version of Vagrant. Therefore vbguest will always support a decent range in "old" Vagrant versions.

Most certainly, this won't make things easier to implement.

features and what didn't happend

When creating vbguest, I thought people would start to contribute installation routines for their favorite OS types. To make this easier, v0.6 got a new feature which allows you to create Custom Installers.
To my surprise, some things (didn't) happened:
People seems to stick with either ubuntu or centos. Till today, debian/ubuntu and centos are the only high level linux distributions with build-in support in vbguest. Also those are the only ones for which bug-fixes and enhancements are regularly requested.
There are infrequent requests on providing support for Windows. But, due to my lack of time and working strictly with unix-like system for almost a decade now, my Windows scripting abilities vanished completely and I depend on the support of people just knowing better than me. Pretty much the same goes for other systems I loved to provide direct support for (like \*BSD, OSX, weirdness of other linux-derivates).

The whole Installers module was designed to be useful in building an own set of installers and even pack and distribute them as a gem or even Vagrant plugin. I've never seen anyone using them that way.

My current interpretation of the apparent lack of use of Custom Installers is, that there is a lack of usefulness and usability for the majority of users in it.

Why is this even important you might ask?

I still believe, that being able to write your own installation routine (and contribute it back into the community if you like) is more helpful than depending on a single project and it's maintainers to build it for you.
Also, it's a good example for a basic design principle I tried to build vbguest after: Being able to have (company) secrets while using an open source project as a foundation to build upon. And again, contribute back to the community if you'd like.
Last but not least, this particular feature makes it harder to comply with Varant's current plugin API.

where to go

I'd like to develop vbguest into a more comprehensive way. This is what should be done to get to a stable 1.0 version:

  • dropping support for Vagrant 1.0 (and below) to allow major overhauls
  • encourage more people to build their own installation routines by providing better ways of doing so. Like an easy-to-use DSL and supporting simple script-execution again.
  • supporting different kinds of "GuestAdditions" like VMware Tools for example.
  • providing better docs. Nobody reads an endless Readme file, right?
  • adding or providing support for other plugins that are out of vbguest's scope (e.g. vagrant-cachier, vagrant-proxyconf, etc.)
  • finally, get some tests running. (esp. integration tests to run against different guests)

Most certainly, vbguest won't deliver a drop-in solution for each and every edge case. It's goal is however, to deliver a easy solution to implement it yourself.

how and when to get there

Vagrant-vbguest is and will continue to be a side project. As such, it's quite impossible to say when things get done.

If you are disappointed to hear this, maybe this will cheer you up:

I'm very excited to announce, that vagrant-vbguest has two new committers. Besides running own open source project and being around in vagrant-world, Anthony aka. @scalp42 and Teemu aka. @tmatilai have been contributing to vbguest for quite some time.

Again, welcome on board! I hope, we'll have a great time in pushing this thing forward and make more people happy.

Keep on nagging

Please, keep on voicing ideas, complaints, bug reports or whatever you think could make vagrant-vbguest a better pice of software. We will try to get back to you as soon as our time permits.

In joyful anticipation of whatever the future will bring, keep on automating!
